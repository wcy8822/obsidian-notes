## 2）三段论据（现状与问题 → 关键洞察 → 路径方案）

### A. 现状与问题（它在解决什么阅读痛点）

- 招商信息天然“要点多、逻辑链长、读者耐心短”，如果用大段文字会直接失焦。
- 这张图把信息压缩成“可扫读块”，每块只承担一个任务：价值、算账、风险、行动节奏、沟通五步法、下一步动作。
- 通过强层级标题与视觉锚点（图标/编号/色条），读者可以不按顺序精读，也能快速定位自己要看的部分。

### B. 关键洞察（它的排版结构长什么样）

我看到的结构（自上而下的叙事链）：

1. **封面区** ：主标题 + 适用对象/版本副标题（“这是给谁用的”先说清）。
2. **价值总括区** ：一条“核心价值”横向强调条（灯泡图标+描边+高对比），作为整张图的“总纲”。
3. **三块底层论证卡** ：商业模式 / 收益模型分析 / 风险识别预警（每块独立卡片，左上图标提示语义）。
4. **行动节奏区** ：用渐变分区标题条引出“快速启动”，下面是按阶段拆分的流程卡，并在右侧用 **圆形编号徽章** 标记顺序（含“3–6”这种合并阶段，减少冗余）。
5. **沟通五步法区** ：同样用分区标题条引出核心方法，内部每一步用 **橙色强调标题+左侧色条/括线** 做强引导；步骤内再用 **绿勾/黄警示** 把“可做/注意”拆开。
6. **底部CTA区** ：三条“下一步行动引导”清单（全绿勾），把读者从理解推到执行。
7. **页脚** ：版本/归属信息弱化呈现，完成闭环但不抢注意力。

### C. 路径方案（可复用的排版方法论：怎么照着做）

**方法论 1：先搭信息骨架，再做视觉皮肤**

- 固定长图骨架： **封面 → 总纲一句话 → 3个论证模块 → 流程节奏 → 方法话术 → CTA** 。
- 每个模块只回答一个问题：
	- 价值：为什么做
	- 算账：能赚多少
	- 风险：会踩什么坑
	- 节奏：怎么启动
	- 话术：怎么讲清楚
	- CTA：下一步做什么

**方法论 2：卡片化 = 降低认知负荷的最小单元**

- 统一使用“圆角卡片 + 内边距 + 轻描边/阴影”的容器体系，让读者天然认为“每块都能独立读完”。
- 卡片内部采用“标题（短）+ 1–3行要点（短句）”，避免段落化。

**方法论 3：颜色语义分工，不用颜色做装饰**

- 暗底：提供沉浸、让高亮元素更突出。
- 橙色：用于“关键步骤/核心结论/标题强调”（驱动注意力）。
- 绿色：用于“已确认可执行项/正向清单”（增强确定性）。
- 黄色：用于“风险/提醒”（把注意力拉回边界条件）。
- 渐变分区标题条：用于“章节切换”，给长图制造节奏感与停顿点。

**方法论 4：视觉锚点三件套：图标、编号、左侧标记线**

- 图标：定义语义类别（价值/收益/风险/行动）。
- 编号徽章：定义顺序与进度（尤其适用于流程）。
- 左侧色条/括线：定义层级与引用范围（让读者知道“这一段属于哪个步骤”）。

**方法论 5：长图节奏控制（防疲劳）**

- 每个大模块之间必须有“分区标题条 + 留白”，形成清晰换气点。
- 用“短卡片串联”替代“长文本滚动”，保证每屏都有结论点。
- 重要内容用“强调条/描边框”做一次“视觉升维”，避免全图同一强度导致无重点。

**方法论 6：从“说明”到“执行”的闭环设计**

- 上半部分解决“信不信”（价值、算账、风险）。
- 下半部分解决“怎么做”（节奏、五步法话术）。
- 最底部解决“现在做什么”（CTA清单），把读者拉出阅读态，进入行动态。

## 1）一句话结论

要让大模型稳定产出“同款排版模板”，关键不是让模型“学风格”，而是把风格抽象成 **可执行的版式语法** ：内容先结构化成标准信息架构（IA），再由模板引擎按规则渲染为组件树（Design Tokens + Layout Rules + Module Library），模型只负责“结构映射与参数填充”，而不是自由发挥。

## 2）三段论据（现状与问题 → 关键洞察 → 路径方案）

### A. 现状与问题

- 直接让大模型“照着图做排版”，输出往往不稳定：一换主题就走样；一加内容就崩版；无法规模化复用。
- 症结在于：你缺的不是“提示词更长”，而是 **可复用的中间表示（IR） **与** 版式约束系统** ，让模型被迫在轨道上生成。

### B. 关键洞察：把“排版”拆成三层可控对象

1. **信息层（What）** ：内容应该被分成哪些模块、顺序是什么、每块解决什么问题。
2. **结构层（How-Structure）** ：每个模块用什么组件结构承载（标题/副标题/要点/图标/徽章/清单）。
3. **样式层（How-Style）** ：颜色语义、字体层级、间距、圆角、阴影等 Token 化。

当你把它变成“信息→组件→样式”的流水线，大模型的任务从“创作”降级为“填表”，可控性立刻上来。

### C. 路径方案：用大模型生成模板的推荐实现路径（从轻到重）

#### 路径 1：先做“模板语法”，再做“自动填充”（最快落地）

**目标** ：产出一个可复用的长图模板（模块库+规则），模型根据输入内容自动挑模块、排顺序、填文案。

- 交付物1： **模板规范（Template Spec）**
	- 模块清单：封面、总纲强调、论证卡组、流程卡组、方法论步骤组、CTA清单、页脚
	- 每个模块的字段：title/subtitle/bullets/icon/badge/notes（长度约束）
	- 排版约束：每模块最大行数、每条 bullet 最大字数、溢出策略（折叠/拆卡/分页）
- 交付物2： **结构化中间层（IR，建议用 JSON/YAML）**
	- 由 LLM 输出：选择哪些模块 + 模块顺序 + 每模块字段内容
- 渲染层：用固定代码（HTML/CSS、PPTX、Figma API 任一）把 IR 渲染成图

> 这条路径的本质：模型输出“组件树”，你用确定性渲染器输出“版式”。

#### 路径 2：模块库 + 规则引擎（适合规模化、多业务线）

**目标** ：不同主题共用同一“排版语法”，只换内容与少量参数。

- 增加一个 **规则引擎** （可以是纯代码或规则表）：
	- 若内容类型=招商/宣讲/复盘 → 推荐模块序列 A/B/C
	- 若数据密度高 → 自动选“表格/算账卡”，并拆成多卡
	- 若风险点≥3 → 自动插入“预警卡”且置于论证区末尾
- LLM 负责：
	- 内容分类、字段抽取、模块匹配（受规则限制）
- 引擎负责：
	- 版式决策的确定性执行（减少漂移）

#### 路径 3：从“图片风格”到“企业设计系统”（最强但投入最高）

**目标** ：把这套长图风格沉淀为 Design System，让生成物可管控、可迭代、可审计。

- 把颜色语义、字体层级、间距栅格、圆角阴影等沉淀为 **Design Tokens**
- 把卡片、分区标题条、编号徽章、勾/警示标记沉淀为 **组件库**
- 大模型只输出：Token 参数 + 组件组合关系
- 优点：可跨产物（长图/海报/单页/PPT）一致复用

20251216\_1910\_一页纸招商与周报自动渲染模板的落地方案

## 1）一句话结论

你要做的是一条“内容到版式”的生产线： **你提供文字 → 大模型抽取成标准 IR（结构化中间层）→ 规则校验与压缩 → 确定性渲染器输出一页纸（PNG/PDF/PPT）** ；模板稳定与可扩展的关键在于“IR+规则+渲染器”，而不是让模型直接画版。

---

## 2）三段论据（现状与问题 → 关键洞察 → 路径方案）

### A. 现状与问题

- 你希望招商、周报长期“长得一样”，这意味着 **版式必须确定性** ；大模型天然随机，直接生成排版会漂。
- 你输入的文字会有长度波动，必须有 **溢出策略** （压缩、拆卡、删冗余、降级展示），否则“模板一页纸”无法长期稳定交付。

### B. 关键洞察

要把“同款一页纸”做成可复用能力，需要三件事：

1. **模板语法（模块库+字段约束）** ：这一页固定由哪些模块组成，每个模块允许多少字/多少条。
2. **结构化中间层 IR（JSON）** ：模型只输出结构化内容（标题、要点、数据、风险、行动等），不碰排版细节。
3. **确定性渲染器** ：用代码按模板把 IR 渲染成成品（PNG/PDF/PPT），保证“每次都像”。

### C. 路径方案（建议按这 6 步做，2 周内可跑通 MVP）

#### Step 1：先定“一页纸信息架构”

你要覆盖两类产物： **招商一页纸、周报一页纸** 。建议都用同一骨架，只换模块开关：

- 顶部：标题 + 副标题（适用对象/周期/版本）
- 核心：一句话结论（总纲强调条）
- 论证区（卡片组）：收益/进展/关键指标、风险、需要支持
- 行动区：下周计划 / 下一步动作（清单）
- 页脚：Owner/日期/版本号

> 招商与周报差异，本质是“论证区字段不同”，版式不变。

#### Step 2：定义 IR（JSON Schema）与长度硬约束

给模型一个“必须填的表”，并规定每个字段上限（这是稳定性的核心）。示例（可直接用）：

要点： **宁可字段少、约束硬，也不要一开始追求“表达自由”。**

#### Step 3：两段式 LLM 生成（抽取 → 压缩对齐）

- **Prompt A：抽取** （把你原始文字变成 IR 草稿，尽量全）
- **Prompt B：排版对齐** （按字数上限“压缩、去重、合并、提升信息密度”，并强制输出合法 JSON）

并在系统侧做校验：

- JSON 是否可解析
- 每字段是否超长
- bullets 条数是否超限
- 若失败：把“报错 + 超限字段”回喂 Prompt B 让模型重写（这是闭环，而非人工改稿）

#### Step 4：选择渲染路线（给你三条可选，按“最稳”推荐）

**路线 R1（推荐）：HTML/CSS 模板 + Puppeteer 导出 PNG/PDF**

- 优点：排版控制力强、迭代快、自动导出图片最方便
- 适合：你要“长得一致 + 自动出图”

**路线 R2：PPTX 模板 + 程序填充（pptxgenjs）导出**

- 优点：团队易编辑、可继续手工微调
- 风险：复杂自适应排版比 HTML 难，溢出处理更麻烦

**路线 R3：Figma 组件库 + API 填充**

- 优点：设计规范最强
- 成本：前期搭建和权限/流程相对重

如果你的目标是“快、稳、自动出图”，优先 R1。

#### Step 5：把“溢出”当成一等公民（否则必崩）

你需要预置 4 个降级策略（按顺序触发）：

1. 句子压缩（同义改写、去形容词）
2. 条目合并（两条合一条）
3. 降级展示（从 bullets 变成一句总结）
4. 模块折叠（低优先级卡片隐藏或移到“附录”——但你要求一页纸时，就直接隐藏）

这些策略应该由规则引擎触发，LLM 负责具体改写。

#### Step 6：验收指标（让它可持续迭代）

- 自动指标：字段超限率、重写次数、溢出次数、生成耗时
- 人工指标：扫读 10 秒能否复述“结论/收益/风险/下一步”
- 版本管理：Template Spec（schema+tokens）与渲染模板必须有版本号，避免“越改越乱”
