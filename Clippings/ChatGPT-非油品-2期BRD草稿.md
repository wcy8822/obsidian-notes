## 1）一句话结论

非油品2期不要再“补洞式交差”，而是把 **可运营的闭环底座** 一次性补齐：以“逆向退款 + 合规链路 + 营销与出资 + 运营可控（数据/风控/履约兜底）”为主线，明确P0边界与联调验收口径，会议目标是当场拍板“做什么/不做什么/谁负责/何时验收”。

## 2）三段论据（现状与问题 → 关键洞察 → 路径方案）

### A. 现状与问题：1期能跑通交易，但不能投入运营

你描述的“1期糊弄交差”，本质是只完成了 **正向链路** （商品创建+交易+结算），但 **逆向、合规、营销可控** 缺失，导致：

- 无法承接真实售后（退款/退货/部分退），运营不敢放量；
- 合同/证照/开票规则不闭环，规模化风险极高；
- 调价与出资能力不足，无法做“价格力/活动”驱动渗透率；
- 缺少监控与异常兜底，SLA与投诉不可控。  
	（你们现有材料里也明确：退款逆向链路缺失是显著空洞。）

### B. 关键洞察：2期的北极星不是“功能数”，而是“可运营性”

2期要以“能不能线上运营、敢不敢放量”为验收标准，因此能力建设必须围绕：

1. **逆向闭环** ：钱、账、票、库存、优惠回滚一致；
2. **合规闭环** ：合同关联、证照链路、亮证、开票规则固化；
3. **营销闭环** ：非油调价/活动能力 + 营销池出资可控可核算；
4. **运营闭环** ：数据口径、监控报警、风控限流、履约异常处理。  
	（你们主线文档也把“退款三键一致、留痕”等作为底线抓手，2期就是把这些从原则变成产品能力。）

### C. 路径方案：用“会前对齐→会上拍板→会后落地”三步推进

**会前（你准备的简版BRD）** ：只写三件事

- 1期现状与不可运营的缺口清单（直指问题）；
- 2期范围（P0必须/ P1可选 / 明确不做）；
- 里程碑与验收口径（DoD）+ 依赖（法务/财务/结算/营销池/数据）。

**会上（60–90分钟）** ：目标是“冻结范围与责任”，而不是讨论设计细节

- 先统一：2期“可运营”定义与P0验收；
- 再逐条拍板：逆向/合规/营销/运营可控 四大包的P0范围；
- 最后锁定：Owner、排期、联调窗口、灰度策略、回滚方案。

**会后（产出一页纪要+行动项）** ：把每个能力拆成“接口/配置/验收数据/风险兜底”。

## 2）三段论据（现状与问题 → 关键洞察 → 路径方案）

### 2.1 现状与问题

- 1期仅满足“商品创建+交易+结算”的正向能力，缺少逆向与合规、营销可控能力，导致无法线上运营放量（售后、票据、补贴核算、风险控制均不可控）。
- 2期若继续“堆功能点”但不定义底线与验收，将再次出现“能上线但不能运营”的结果。

### 2.2 关键洞察

- 2期的业务目标不是“做完功能”，而是“ **具备可运营资格** ”：能够承接真实售后、满足合规展示与留痕、支持价格力与活动、并能在异常时快速止损。
- BRD要明确“做什么、做到什么程度算完成（DoD）”，而不是展开PRD级实现细节。

### 2.3 路径方案

- 以四个能力包组织2期范围： **逆向闭环 / 合规闭环 / 营销闭环 / 运营可控** 。
- 为每个能力包给出：业务定义、边界、底线要求、验收口径（可测试、可对账、可追溯）。
- 协同上不搞“财税法安全一起大拉会”，改为： **产品主线推进 + 必要时按条目异步确认/卡点评审** ，减少组织摩擦。

## 3）BRD正文（做什么能力 + 合规底线 + 验收口径）

### 3.1 项目概述

- **项目名称** ：非油品线上化（随手购）2期能力建设
- **建设目标** ：从“能跑通”升级到“可运营、可灰度、可放量”
- **适用范围** ：非油品订单（与油品同单或关联交易场景）

### 3.2 业务目标（Outcome，不写实现）

1. 支持非油品售后全流程：可部分退、可SKU级退，资金/账务/票据一致
2. 合规链路闭环：合同与证照齐全、可亮证、开票规则稳定可审计
3. 具备基础营销与价格力：非油可调价、支持营销池出资并可核算追溯
4. 运营可控：可灰度、可开关、可监控追责

### 3.3 2期范围（P0必须 / P1建议）

#### P0（必须交付，否则仍不可运营）

**A. 逆向能力（Refund & Reverse）**

- **A1 单独退非油品（订单级部分退款）**
	- 定义：同单场景下仅对非油部分发起退款/退货，不影响油品链路
	- 边界：仅支持“非油退、油不退”；不改造油品既有逆向规则
	- 验收（DoD）：
		- 可发起、可完成、可追踪状态（申请/审核/完成/失败原因）
		- 退款资金原路退一致；对账侧可看到负向冲回记录（不要求在BRD写到字段级）
- **A2 单独退SKU（SKU级部分退/部分数量退）**
	- 定义：一笔非油订单内多SKU可按SKU/数量维度退款
	- 验收（DoD）：
		- SKU级退款后订单金额、优惠分摊、应退金额计算正确且可解释
		- 退款后结算可冲回到对应明细（至少可被财务对账识别）
- **A3 营销/出资的逆向回滚（底线项）**
	- 定义：若订单使用营销池出资/商户让利/平台补贴，退款需同步冲回并留痕
	- 验收（DoD）：
		- 退款后营销池消耗可回滚或形成可核算的负向记录
		- 任何退款都能追溯“优惠来源与分摊逻辑”（满足审计与争议处理）

**B. 合规能力（Compliance）**

- **B1 合同关联（底线）**
	- 定义：非油业务必须绑定“签约主体/站点/商户”的合同关系，缺失则不可交易或不可结算（由策略决定）
	- 验收（DoD）：
		- 可校验“是否已关联有效合同”；无合同的订单可被阻断或被标记为不可结算（策略二选一，会上拍板）
- **B2 证照上传与管理（底线）**
	- 定义：支持商户/主体上传必需证照，包含有效期管理；缺失/过期触发交易限制策略
	- 验收（DoD）：
		- 证照可上传、可查看、可更新；到期状态可识别
		- 到期/缺失可触发站点级或商户级“限制开关”（不写具体UI）
- **B3 亮证（底线）**
	- 定义：C端展示“经营主体与资质信息/售后责任信息”（以监管要求与用户知情为导向）
	- 验收（DoD）：
		- 下单页/订单页可展示必要主体信息与证照入口（最小满足监管/消费者知情）
- **B4 B票规则固化（底线：写死不调）**
	- 定义：
		- 油品：严格沿用既有flow规则（不改、不碰）
		- 非油：按B票规则固化执行，不开放运营侧配置，避免合规漂移
	- 验收（DoD）：
		- 非油订单触发的开票规则稳定一致，可追溯且可解释（含退款对应的票据处理路径）

**C. 营销与调价能力（Pricing & Promotion）**

- **C1 非油调价能力**
	- 定义：支持对非油SKU进行调价（至少覆盖站点/SKU维度；是否需要时段由资源决定）
	- 验收（DoD）：
		- 调价可生效、可回滚、可追溯（生效范围与时间）
		- 不影响油品价格体系与油品活动链路
- **C2 非油支持营销池出资**
	- 定义：非油活动支持从营销池出资（可与商户让利叠加规则需明确）
	- 验收（DoD）：
		- 可记录每单出资金额与归属来源
		- 可汇总统计（预算消耗、订单覆盖、净补贴），并与退款回滚一致

**D. 运营可控（Operate & Control，P0最小集）**

- **D1 灰度与开关**
	- 定义：支持站点/商户维度的功能开关（至少覆盖：非油售后、非油营销、非油交易入口）
	- 验收（DoD）：
		- 可按站点灰度上线；异常可快速关闭，不影响油品链路
- **D2 关键链路监控（最小看板/告警）**
	- 定义：对下单/支付/履约/退款/开票/结算关键节点提供最小监控与异常定位能力
	- 验收（DoD）：
		- 可按日查看成功率与失败原因Top；能定位到订单级问题用于排障

#### P1（建议项：视资源与放量节奏决定）

- 库存/超卖治理（若2期就要强运营放量，建议提前）
- 更复杂营销玩法（券/满减/组合包）
- 商户侧自助对账/售后处理能力增强

### 3.4 明确不做（2期范围外，防止发散）

- 不改造油品正向/逆向/开票主链路
- 不做大型客服工单系统重建（仅要求最小可定位字段与查询能力）
- 不开放B票规则为运营配置项（保持“写死不调”）

### 3.5 安全合规底线要求（不拉大团队，但必须写清楚）

> 这里是“底线条款”，任何上线必须满足；不讨论实现方式。

1. **主体与资质可追溯** ：每笔非油订单必须能追溯到签约主体、站点、证照状态与有效期
2. **亮证可达** ：用户侧可查看必要经营信息与证照入口（满足监管与知情）
3. **票据规则稳定** ：油品不动；非油B票规则固定执行，不允许运营侧动态配置
4. **逆向一致性** ：退款必须同步反映到资金、结算、补贴出资（至少保证“可核算、可追溯、不丢账”）
5. **可止损** ：必须具备站点级灰度与快速关闭能力，异常不扩散到油品链路

### 3.6 验收口径（建议你在会上用来“卡住交付质量”）

- **功能验收** ：A/B/C/D四包的P0条目逐项验收通过
- **一致性验收** ：随机抽样订单（含：无优惠/有优惠/有营销池出资/多SKU/部分退），核对“应退=实退=结算冲回=补贴冲回”
- **合规验收** ：无合同/证照缺失/证照过期三类场景策略可触发且可解释；亮证入口可达
- **灰度验收** ：指定站点上线、指定站点关闭，链路不串台、不影响油品

## 4）与产品团队的协同方式（相互补位，不把财税法安全一起拉进来）

### 4.1 分工原则（RACI简化版）

- **你/业务侧（非油品运营&业务Owner）**
	- 负责：业务规则拍板（边界、优先级、口径）、验收用例、灰度策略、运营SOP
	- 输出：本BRD、验收清单、样例订单与对账校验脚本需求（不写实现）
- **产品侧（产品Owner）**
	- 负责：需求拆解成可交付迭代、范围管理、防止发散、跨系统协同对齐（但不组织“大拉会”）
	- 输出：PRD/交互、研发排期、联调计划、上线与回滚方案
- **研发/测试**
	- 负责：交付与质量、联调、自测覆盖、灰度发布与监控接入

### 4.2 “不大拉会”的合规协同策略（建议写进会议纪要）

- **按条目异步确认** ：合同关联、证照必传项、亮证文案、B票规则这四类，用“单条需求卡点评审”方式分别确认，而不是把财务/税务/法务/安全一次性拉齐开大会。
- **卡点机制** ：
	- 卡点1（需求冻结前）：底线条款确认完成
	- 卡点2（提测前）：抽样用例跑通（含逆向+出资回滚+亮证）
	- 卡点3（灰度前）：开关与回滚演练完成

### 4.3 会议与节奏（建议）

- 每周一次30分钟“范围与风险例会”（仅你+产品+研发TL）
- 关键节点（需求冻结/提测/灰度）各一次15分钟“验收对齐会”
- 其余合规问题全部走单点评审与异步留痕，避免多方同时involve导致排期失控
