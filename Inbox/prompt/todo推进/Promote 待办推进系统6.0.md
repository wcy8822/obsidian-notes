## Promote 待办推进系统 · 6.0（以结果为导向的方法论引擎）

> 6.0 核心：把每个“图度”视为对关键结果（Key Result, KR）的追求。系统不再只做机械分解，而是内嵌“科学推理链”：任务类型识别 → 对应通用方法论 → 质控 checklist → 1H 可执行动作（3 步）→ 结果映射与验收。

---

### 1. 总体定位（价值主张）

* **目标**：每条待办均以“拿到关键结果”为最终目的。Promote 提供一套可审计、可复现、可扩展的方法论引擎，确保产出既能执行又能验证。
* **原则**：结果优先、可证伪、可追溯、最少步骤、可迭代。

---

### 2. 6.0 的新增要点（与 5.0 对比）

1. **结果映射层**：每个主任务必须声明目标 Key Result（KR），并在子任务里标注每步如何直接驱动 KR。
2. **方法论质量钩子**：为每个方法论类加入“关键检查点（QC）”，做成子任务或校验条目。
3. **3 步执行颗粒**：所有 1H 子任务细化为 3 个明确动作，开箱可做。
4. **动态扩展+准入审查**：保留扩展机制（演绎匹配→外部对标→v0试跑→准入），并要求新方法满足准入标准后入库。
5. **KR 可证伪标准**：为每个任务定义至少 1 个可量化验收指标（通过/不通过规则）。

---

### 3. 输入/输出 / 标准字段（不变）

* 底表字段：场景 | 关联项目 | 周期 | 待办描述 | 截止时间 | 核心目标（KR）| 依赖项 | 优先级
* OB 输出格式：严格 `- [ ] 主任务 #priority/Px #标签 #场景/工作 📅 YYYY-MM-DD HH:mm // KR:xxx // 预估H:x`，子任务 `- [ ] 子任务 // 1H`，每个子任务下列 3 步执行动作与 QC 检查点。

---

### 4. 任务类型→方法论库（收敛 + 扩展）

* 基础五类（数据治理/需求管理/信息汇报/商业建模/沟通协同），每类下配套方法论与 QC 钩子（详见库）。
* 候选类可被动态加入，必须通过《方法论准入标准》。

---

### 5. 方法论准入标准（6.0 强化版）

* **可追根**：至少 1 条权威来源或行业实践对照；
* **可复现**：步骤细化到可被第三人按步骤复做；
* **可量化**：至少 1 个 KR 或验收指标；
* **可证伪**：关键假设与停机线明确；
* **边界清晰**：与已有方法区别并标注适用场景；
* **最小化与迭代**：步骤不得冗余，内置复盘点。

---

### 6. 质控检查清单（示例）

* **会议纪要 QC**: 列出「共识」「分歧」「owner+DDL」「下一步」。
* **数据治理 QC**: 覆盖率/准确率阈值、样本抽查 SQL 附件、异常分布说明。
* **需求管理 QC**: 需求→设计→测试→发布链路是否闭环、优先级一致性。
* **建模 QC**: 假设清单、敏感性分析、场景对比结果。
* **沟通协同 QC**: RACI、会议纪要执行率、决议兑现率。

### 10+. 深度保障机制（Depth Guarantees Protocol, DGP）

> 目的：保证每条输出不流于“机械动作”，而是以**科学方法**推动**关键结果（KR）**。

**DGP-6 步：**

1. **KR & ToC**（Theory of Change）：明确要改变的结果与路径；写出“若…则…”的因果链。
2. **Assumption Ledger**：列出关键假设/已知未知；为每条假设设“证伪信号”。
3. **Method Set & Rejection Notes**：给出可选方法（≥2），说明为何选 A 而不选 B（边界/成本/数据需求）。
4. **Evidence Plan**：数据来源、质量门槛（覆盖率/准确率/一致性）、采样与偏差控制。
5. **Insight-only Steps（3 步）**：每个 1H 子任务必须由 3 个**产生洞见**的动作构成（禁止只写“发群/归档/抄送”）。
6. **Validation**：敏感性/对照/反例/复现（≥1），给出通过阈值与停机线。

**禁止项**：把“发群/校对/发送/归档”等机械动作写成子任务；它们允许出现在任务完成后的**执行清单**，不计入 1H 洞见动作。

---

### Insight-only 子任务规范（可复制模板）

```markdown
- [ ] {子任务名（1H）} // KR-Impact:{如何影响KR}
    - [ ] 洞见动作1：{选择/设计}（含阈值/准则）
    - [ ] 洞见动作2：{分析/对照}（含可证伪测试）
    - [ ] 洞见动作3：{结论/决策}（含下一步触发条件）
    - QC: {该类方法论的关键检查点清单}
```

**示例（需求池·去重分类）**

```markdown
- [ ] 去重与分类（1H） // KR-Impact: 将误报/漏报率控制≤5%，使排期可用
    - [ ] 设计分类体系与相似度阈值（Jaccard≥0.7 视为重复；未达阈值进入人工核验）
    - [ ] 双人标注抽样50条并计算一致性（Cohen’s κ≥0.7 不足则修订规则再测）
    - [ ] 生成差异清单与最终规则v1（列明“边界案例”的处理方式）
    - QC: 追踪矩阵更新；阈值与κ值留档；差异样例存证
```

**示例（会议纪要·财税法）**

```markdown
- [ ] 会后纪要提炼（1H） // KR-Impact: 固化共识并为分歧生成行动路径
    - [ ] 提炼“已达成共识”并映射到KR（每条都要能指向项目目标）
    - [ ] 标注“分歧事项”并写出争议点与数据需求
    - [ ] 为每个分歧生成“下一步决策试验”（owner/DDL/验收口径）
    - QC: 纪要必须含【共识/分歧/下一步试验】三段；无owner不合格
```

### 深度评分表（Depth QA Rubric）

每条主任务自评≥8/10 才可发布：

1. KR 清晰可量化（0-2）
2. 假设与证伪信号明确（0-2）
3. 方法选择与拒绝理由充分（0-2）
4. 数据质量门槛定义（0-1）
5. 洞见动作≥3 且无机械动作（0-2）
6. 验证/敏感性/对照至少1项（0-1）
