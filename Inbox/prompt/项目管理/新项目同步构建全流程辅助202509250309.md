# 2025-09-25 星期四 Daily

### 提示词：新项目「代码开发+文档系统」同步构建全流程辅助

**目标**：从项目启动到迭代维护，全程保障代码与文档同步，避免后期返工，实现“代码即文档，文档即代码”的协同闭环。

**使用场景**：新项目启动时，向AI（推荐`claude-4-sonnet MAX`/`GPT-4`）输入项目核心信息，获取全流程指导与初始产出。


### 需替换的项目核心信息（请先补充）
- 项目名称：《____系统/模块》
- 核心业务目标：____（如“构建高可用的门店标签计算系统，支持实时推导与配置化管理”）
- 技术栈：____（如“Python 3.11 + MySQL + FastAPI + Docker”）
- 预期用户/使用方：____（如“算法团队调用接口，运维团队负责部署”）


### 提示词正文
“请作为「项目全流程协同顾问」，辅助我从0到1启动新项目，**让代码开发与文档系统同步构建**，确保前期规划、开发过程、后期维护中，代码逻辑与文档描述完全一致，最大程度减少返工。请按照以下4个阶段，结合我提供的项目核心信息，输出具体指导与初始成果：


### 阶段1：项目启动前 · 预设计「文档+代码」协同框架
1. **输出「文档系统核心框架」**：
   - 基于业务目标，规划文档核心模块（如`00_项目总览`「含背景、架构图」、`01_开发指南`「含环境搭建、规范」、`02_配置管理`「含所有配置文件说明」、`03_业务逻辑`「各核心功能的逻辑+代码关联」、`04_部署与运维`「含Makefile/脚本/环境要求」、`05_迭代日志`「记录版本变更」）；
   - 每个模块需明确**“必须关联的代码文件类型”**（如`02_配置管理`需关联`config/*.yaml`、`src/config_loader.py`）。

2. **输出「代码目录规范」**：
   - 设计匹配技术栈的代码目录结构（如`src/`下分`model/`「数据模型」、`service/`「业务逻辑」、`api/`「接口层」；`config/`存配置文件；`scripts/`存部署/测试脚本）；
   - 对关键目录（如`src/tag_derive/`「核心业务模块」），标注**“需同步维护的文档章节”**（如`src/tag_derive/`需同步更新`03_业务逻辑/标签推导.md`）。

3. **输出「协同约束清单」**：
   - 列出**“代码提交前必须同步更新的文档内容”**（如“新增配置项时，需同步更新`config/`下的配置说明文档 + `02_配置管理`对应章节”）；
   - 明确**“文档与代码的双验证规则”**（如“每段核心代码逻辑（函数/类），必须在对应文档中说明，且文档需标注「代码路径+行号」”）。


### 阶段2：开发过程中 · 代码与文档「同步编写」
假设我正在开发**「标签推导核心模块（`src/tag_derive/derive.py`）」**，需实现“从配置文件读取规则，计算门店标签”的逻辑。请模拟此场景，输出：

1. **「代码编写辅助」**：
   - 生成该模块的**初始代码框架**（含关键函数、配置读取逻辑，需匹配技术栈）；
   - 在代码中强制添加**注释要求**：函数文档字符串需说明「输入参数、输出结果、配置依赖」，且标注**对应文档章节**（如“`def calc_store_tag():` 的文档字符串需标注「对应`03_业务逻辑/标签推导.md#规则计算`」”）。

2. **「文档同步模板」**：
   - 生成`03_业务逻辑/标签推导.md`的**对应章节模板**，包含：
     - 核心逻辑描述（需与代码中函数逻辑完全一致）；
     - 配置依赖说明（关联`config/tag_rules.yaml`的结构示例）；
     - 代码关联标注（如“代码路径：`src/tag_derive/derive.py` → 核心函数：`calc_store_tag()`”）；
     - “待验证点”（如“需验证与`config/tag_rules.yaml`中`rule_type=1`的匹配逻辑”）。

3. **「双检提醒」**：
   - 输出开发此模块时，需同步检查的**“文档+代码”事项**（如“写完`calc_store_tag()`后，需检查：① 函数注释是否与文档章节描述一致；② `config/tag_rules.yaml`的示例是否覆盖代码逻辑；③ `03_业务逻辑`文档是否标注代码路径”）。


### 阶段3：验证与优化 · 文档-代码「一致性校验」
当完成一个功能模块（如“标签推导”）后，需验证文档与代码的一致性。请输出：

1. **「一致性校验清单」**：
   - 列出需校验的维度（如“配置项名称是否一致、函数入参/出参与文档描述是否一致、代码中异常分支是否在文档中说明”）；
   - 提供**“快速校验方法”**（如“用代码注释中的文档章节链接，跳转检查文档内容；用文档中的配置示例，代入代码跑单元测试”）。

2. **「问题修复指引」**：
   - 模拟“代码中配置读取逻辑与文档说明不一致”的场景，输出**修复步骤**（如“先修正代码逻辑 → 同步更新`config/`下的配置说明 → 修改对应文档章节 → 在`05_迭代日志`中记录变更点”）。


### 阶段4：持续维护 · 迭代与文档「同步演进」
当项目进入迭代（如新增“标签批量更新”功能），需确保文档同步演进。请输出：

1. **「迭代前准备」**：
   - 输出**“迭代影响评估清单”**（如“新增功能需修改哪些代码模块？对应哪些文档章节？是否涉及配置文件变更？”）。

2. **「迭代中同步」**：
   - 生成**“迭代日志模板”**（如`05_迭代日志/202X-XX-XX_标签批量更新.md`），要求包含：
     - 代码变更点（如“`src/tag_derive/`新增`batch_derive.py`”）；
     - 文档变更点（如“`03_业务逻辑/`新增「标签批量更新」章节，`02_配置管理/`新增`batch_rule.yaml`说明”）；
     - 验证要求（如“需验证`batch_derive.py`与新文档章节的逻辑一致性”）。

3. **「长期维护建议」**：
   - 输出**“文档-代码协同的长期规范”**（如“每季度做一次全量校验；新人接入时，先学习「文档-代码协同约束清单」；代码评审时，需同步检查对应文档是否更新”）。


请基于上述4个阶段，结合我提供的项目核心信息（项目名称、业务目标、技术栈、用户方），生成具体的指导内容与初始文件模板。”


### 使用说明
将提示词中“需替换的项目核心信息”补充完整后，直接输入到AI工具中（推荐`claude-4-sonnet MAX`处理“长上下文+多文件关联”，或`GPT-4`处理“代码细节+结构化输出”），即可获得针对性的全流程协同方案，从项目启动就保障代码与文档同步，减少后期返工。
